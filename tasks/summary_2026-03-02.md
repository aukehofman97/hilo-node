## What was built — V2

### Two-tier event architecture
Events are no longer copied in full to every node. The originating node stores full Turtle triples; peers receive a lightweight **notification** containing only metadata and a `data_url` pointing back to the source. Full triples are fetched on demand via authenticated HTTP.

- `POST /events` now publishes an `EventNotification` (event_id, type, subject, data_url) to the queue — not the raw triples
- `store_notification()` in `graphdb.py` writes peer events as metadata-only records with `hilo:dataUrl`
- `GET /events/{id}` returns full triples for local events; for received notifications it returns `links.data` pointing to the source node

### JWT inter-node authentication
Nodes authenticate to each other using RS256 JWTs signed with their private RSA key.

- Each node generates its own RSA keypair at startup (`data/node.key` + `data/node.key.pub`)
- `/.well-known/hilo-node` exposes the node's public key, ID, name, and base URL
- `GET /connections/{peer}/token` issues a fresh JWT for a named peer (signed by this node, audience = peer's node_id)
- `GET /events/{id}` validates incoming Bearer JWTs against the peer's stored public key
- Internal key bypass (`Bearer dev`) allows the local UI to access protected endpoints without a peer token

### Connection management
Nodes establish explicit trust relationships before exchanging data.

- `POST /connections/request` — Node B sends its identity to Node A to request a connection
- `POST /connections/{id}/accept` — operator accepts; triggers acceptance callback to the requesting node
- `POST /connections/accepted` — callback that activates the connection on the initiating node
- `POST /connections/outgoing` — records that this node sent a connection request (fixes acceptance 404 bug)
- `GET /connections` — lists all connections with status (pending_incoming, pending_outgoing, active, rejected)
- `GET /connections/{peer}/token` — generates a JWT for an active peer

### Connections UI page
Full peer management interface built in `ui/src/pages/Connections.tsx`.

- Enter a peer's base URL → preview their identity (fetched from `/.well-known/hilo-node`)
- Send a connection request with one click; records the outgoing state locally
- Connection list with status badges, initiated-by indicator, and bearer token display per active peer
- Token display truncated to first 32 chars with copy-to-clipboard

### Events Monitor — full implementation
`ui/src/pages/Events.tsx` rebuilt from stub.

- Auto-refreshing event table (every 10s) with filter chips (status, type, source node, search)
- Expandable detail panel: metadata + syntax-highlighted Turtle triples
- **Status badge is contextual** — `published` (purple) when you originated the event; `received` (blue) when it came from a peer. Derived dynamically from `source_node` vs `/.well-known/hilo-node`, never hardcoded
- **Cross-node fetch**: when viewing a received notification, a "Fetch full event from {source_node}" button appears. On click: gets JWT from `/connections/{peer}/token`, calls `data_url` on the source node with `Authorization: Bearer {token}`, renders full triples in the panel

### Dynamic TopBar node identity
`TopBar.tsx` fetches `/.well-known/hilo-node` on mount and displays the live node ID instead of hardcoded "node-a".

### Node B — local two-node setup
Second fully independent node running alongside Node A.

- Fuseki dataset `hilo-b` (same Fuseki instance, separate dataset)
- Node B API on port 9000, Node B UI on port 3001
- Separate SQLite DB (`data/node-b.db`) and RSA keypair (`data/node-b.key`)
- Consumer started with `HILO_NODE_ID=node-b` and `HILO_API_URL=http://localhost:9000`
- Consumer on Node A fetches active peers via `GET /connections`, forwards notifications to each peer's `/bridge/receive`

### Consumer forwarding
`queue/consumer.py` now implements peer forwarding.

- On receiving a queue message, fetches `GET {api_url}/connections` for active peers
- POSTs `EventNotification` to each peer's `/bridge/receive`
- Two-tier: the notification has no triples — only enough to identify the event and where to fetch it

## End-to-end flow (verified)

1. POST an event to Node A → stored with full triples in `hilo` Fuseki dataset
2. Consumer picks up the notification from RabbitMQ → forwards to Node B's `/bridge/receive`
3. Node B stores the notification (no triples) with `hilo:dataUrl = http://localhost:8000/events/{id}`
4. Node B's Events Monitor shows the event with status **received** (blue badge)
5. Click the event → detail panel → "Fetch full event from node-a" button
6. Click fetch → `GET /connections/node-a/token` → RS256 JWT issued by Node B, audience = node-a
7. JWT sent to Node A's `GET /events/{id}` → verified against Node B's stored public key → full Turtle triples returned
8. Detail panel renders the complete RDF payload
