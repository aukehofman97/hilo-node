## Findings

### Critical (must resolve before planning)

**C1 — SQLite has no persistent volume**

The architecture introduces a SQLite database for connection state but doesn't specify where
the file lives or how it persists across container restarts. The API runs in Docker. If the
container restarts (which `restart: unless-stopped` guarantees will happen), and the SQLite file
is inside the container filesystem, **all connection state is lost** — every peer relationship
gone, every stored public key gone.

Fix needed: specify `HILO_DB_PATH` (default: `/data/hilo.db`) and add a named Docker volume
`hilo-api-data` mounted at `/data` in `docker-compose.yml`. This is one line in compose and
one config value — but it must be in the architecture or it will be missed in planning.

---

**C2 — `subject` field: no extraction method defined**

`EventNotification` includes a `subject: str` field described as "primary RDF subject URI from
triples." But the triples field is a raw Turtle string. There is no defined algorithm for
extracting "the primary subject" from arbitrary Turtle. Turtle can have multiple subjects.
Without a spec, this is unimplementable — the developer will guess.

Two clean options:
- **Option A**: Add `subject: str` to `EventCreate` — the caller explicitly provides the
  primary subject URI. Simple, no parsing required. Callers already know their primary subject.
- **Option B**: Extract the first subject triple from parsed Turtle using rdflib.
  Works but adds complexity and can still fail for ambiguous graphs.

Recommendation: **Option A**. The caller knows what the event is about. Make them say it.

---

**C3 — Acceptance callback: Node B never receives Node A's public key**

The API surface lists `POST /connections/{id}/accept` as a local operator action. But after
Node A accepts, Node B has no mechanism to find out. There is no callback endpoint listed for
Node A to call on Node B, and no polling mechanism described.

The connection handshake is incomplete as specced. For Node B to use the connection (sign JWTs
that Node A will verify), Node B needs Node A's public key. For Node A to verify Node B's JWTs,
Node A needs Node B's public key (provided in the request — that part works). But Node B never
gets Node A's key.

Fix: add `POST /connections/accepted` to the API surface — a callback endpoint both nodes expose.
When Node A accepts, it POSTs `{status: "accepted", node_id, public_key}` to
`{peer_base_url}/connections/accepted`. Node B receives this, stores Node A as an active peer
with Node A's public key, and the connection is live on both sides.

Failure mode to address: what if Node B is unreachable when Node A tries to call back?
Recommendation: Node A retries the callback 3 times with backoff, then marks the connection
`accept_pending` (accepted on A's side, callback not yet delivered). When Node B comes back
online and fetches `GET /.well-known/hilo-node` from Node A, it can also poll
`GET /connections/{request_id}/status` — or the Connections UI on Node B shows a "refresh"
button to re-fetch.

### Concerns (should address)

**W1 — `data_url` default will break cross-node scenarios**

`HILO_NODE_BASE_URL` defaults to `http://localhost:8000`. In V2 (two containers on the same
machine via docker-compose), this works because Node B can reach Node A on localhost. But if a
developer forgets to set this env var when running Node B, the `data_url` in notifications from
Node A will say `http://localhost:8000/events/...` — which Node B will correctly resolve.
However, even in V2, if the two nodes are on different docker-compose networks (future), or if
someone tests V3 early, this will silently produce unreachable data URLs.

The architecture should call this out explicitly: **`HILO_NODE_BASE_URL` is not optional —
it must be set to the publicly reachable address of this node.** Document in `.env.node-a`
and `.env.node-b` with correct values.

---

**W2 — `/bridge/receive` auth posture not stated**

`POST /bridge/receive` accepts incoming `EventNotification` from peer nodes. The architecture
doesn't state whether this endpoint is authenticated. If it's open, a malicious actor can flood
Node B with fake notifications for data that doesn't exist. If it requires JWT auth, it means
Node B must verify the sender's JWT — which requires an active connection to that peer.

For V2, open (unauthenticated) is probably fine — the notification contains no sensitive data,
and a forged notification will simply result in a 404 or auth failure when Node B tries to
fetch the data. But this should be explicitly stated in the architecture to prevent the
developer from adding unnecessary auth.

Recommendation: state clearly — `/bridge/receive` is intentionally unauthenticated in V2.
Add a note for V3/V4: sender JWT can be added for verification.

---

**W3 — GraphDB dual-write creates an unnecessary coupling for V2**

The architecture proposes writing accepted connections to GraphDB ("as RDF triples, hilo:trusts
relationship") in addition to SQLite. For V2 with 1-2 peers, this adds a GraphDB dependency to
the connection flow — if GraphDB is unhealthy during a connection accept, does the accept fail?

The semantic graph value is real (queryable trust relationships), but for V2 it's premature.
SQLite-only is simpler and doesn't couple connection management to GraphDB health.

Recommendation: drop the GraphDB dual-write for V2. Add it as a planned enhancement for V3/V4
when the network topology becomes interesting to query semantically.

---

### Suggestions (nice to have)

**S1 — `subject` in EventCreate should be an explicit required field**

See C2. If Option A is chosen (caller provides `subject`), this should be documented as a
**breaking change** to the existing `POST /events` API. Any existing callers (curl examples,
test scripts, the README quickstart) will need to be updated.

**S2 — Consumer retry for bridge HTTP calls should be specified**

The consumer currently retries RabbitMQ acks. It will now also make outbound HTTP calls to
peer nodes' `/bridge/receive`. What happens if Node B's `/bridge/receive` returns 500 or times
out? Should the message be re-queued? Dead-lettered? The architecture is silent on this.
Recommendation: specify that a failed bridge delivery (after N retries) moves the notification
to the dead-letter queue, same as other failures.

---

### What's Solid

**V4 upgrade path is clean and realistic.** The `verify_token()` function is the only thing that
changes between V2 (config key lookup) and V4 (DID resolution). The JWT format, the Bearer
header, the `.well-known` endpoint — all carry forward unchanged. This is the right way to
design for a known future state.

**`.well-known/hilo-node` is the correct pattern.** Borrowing from OpenID Connect Discovery
means the base URL is the identity. Manual URL entry for V2 + this endpoint makes peer discovery
trivially upgradable to a registry in V3/V4.

**RS256 over HS256 is the right choice.** No shared secrets between parties. Each node's
private key stays on that node. The key distribution (public keys only) is a solved problem.
This is genuinely better than the alternative and not over-engineered.

**Three-PR structure.** Backend → UI → Node B is the right sequence. Each PR is independently
shippable and testable.

**Lazy data fetch (Node B fetches on demand) correctly embodies the "data at source"
principle** stated in the architecture PDF. Eagerly duplicating all data to Node B would
undermine the whole design.

## Verdict

**Status**: Approved with changes

**Blocking issues before planning**:
- C1: Add named Docker volume for SQLite persistence to architecture doc
- C2: Choose extraction method for `subject` — recommendation is Option A (explicit field in EventCreate)
- C3: Add `POST /connections/accepted` callback endpoint to API surface; define failure mode

**Recommended non-blocking changes** (can fix in planning phase):
- W2: State `/bridge/receive` is intentionally unauthenticated
- W3: Drop GraphDB dual-write for V2 (SQLite only)
- S2: Specify consumer retry behavior for failed bridge deliveries

**Recommended action**: Fix C1, C2, C3 in `tasks/architecture.md`, then proceed to Gate 3 (planning skill).
