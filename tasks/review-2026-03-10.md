## Findings

### Critical (must fix before proceeding)
- None

### Concerns (should address)

1. **`sentry_sdk.set_tag` placement**: The architecture says to add `sentry_sdk.set_tag("node_id", settings.node_id)` after `sentry_sdk.init()` — but this requires importing `settings` in `main.py`. Currently `main.py` doesn't import `settings` at the module level (it imports it inside the lifespan function). The architecture should state explicitly that `from config import settings` must be added to the top-level imports of `main.py`.

2. **`sentry_sdk` import in `events.py`**: The architecture calls for `sentry_sdk.capture_exception(exc)` in `events.py`, but doesn't mention that `import sentry_sdk` must be added to that file. Minor but should be explicit to avoid implementation ambiguity.

3. **`/sentry-debug` removal**: Architecture says remove the route from `main.py`. It was added in this same session and hasn't been committed — should also verify there are no references to it in tests or docs.

### Suggestions (nice to have)

1. **`queue/consumer.py` not audited**: The queue consumer runs as a separate container and has its own error paths (message processing failures). The architecture scopes only the API container — this is reasonable for now, but worth flagging that consumer logging is a separate future task.

2. **`profile_session_sample_rate=1.0` in production**: Profiling 100% of sessions will add overhead at scale. Low risk for now (single-node, low traffic), but worth revisiting if load increases.

### What's Solid

- Using standard `logging` as the interface throughout and relying on `enable_logs=True` for Sentry forwarding is exactly the right call — it keeps Sentry coupling out of business logic files.
- The "What We Are NOT Logging" section is disciplined and prevents noise. Explicitly listing non-log-worthy paths is good practice.
- `capture_exception()` only for the queue swallow case is precisely scoped — avoids the antipattern of wrapping every try/except with Sentry calls.
- Removing the debug endpoint is correctly called out as a production-safety requirement.
