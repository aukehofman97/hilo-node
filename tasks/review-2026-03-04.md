## Findings

### Critical

**C1 — Partial failure creates silent data inconsistency**

The endpoint does two writes:
1. SPARQL INSERT: add `hilo:triplesPayload` metadata to event subject
2. `insert_turtle(triples)`: insert the actual RDF triples into the dataset

If (1) succeeds and (2) fails: the event will report `has_local_copy: True` in the list but the actual triples are missing from the store. The user believes they imported successfully, but a SPARQL query returns nothing.

**Fix — swap the order**: insert triples first, then update metadata. If (1) fails, no metadata is written and the user can retry cleanly. If (2) fails after (1), the user retries: Fuseki re-insertion is idempotent, and (2) now succeeds. Worst case: orphaned triples with no event linkage — far better than a false `has_local_copy` flag.

---

### Concerns

**C2 — SPARQL `OPTIONAL` in `get_events` could return full Turtle payload for every row**

Adding `OPTIONAL { ?event hilo:triplesPayload ?tp }` to the `get_events` SELECT will bind the full Turtle string (potentially hundreds of characters) for every imported event in the list. The architecture says "set `has_local_copy`" but doesn't specify how.

**Fix**: Use `BIND(BOUND(?tp) AS ?hasLocalCopy)` so the list query returns only a boolean, not the payload:
```sparql
OPTIONAL { <...> hilo:triplesPayload ?tp . }
BIND(BOUND(?tp) AS ?hasLocalCopy)
```
This must be explicit in the implementation spec to avoid an inadvertent performance regression.

**C3 — Missing response code for "not a peer notification"**

The endpoint behaviour is currently:
- 404 if event not found
- 409 if already imported

But what if someone calls `POST /events/{id}/import` on a locally-originated event (has `hilo:triplesPayload`, no `hilo:dataUrl`)? The architecture doesn't define this. It should return **400 Bad Request** with a clear message: `"This event was originated locally — import is only valid for peer notifications"`.

The check sequence should be:
1. 404 — event not found
2. 400 — event found but not a peer notification (no `hilo:dataUrl`)
3. 409 — peer notification already imported
4. 200 — proceed

**C4 — "Store locally" button lifetime is session-scoped but not stated**

The button appears only after a successful remote fetch in the current UI session. If the user fetches, closes the detail panel, and reopens the same event, the fetched triples are gone from React state (`detail` is re-loaded from the local API, which still shows a notification). The user must fetch again before they can store.

This is acceptable UX, but it must be documented so the implementation handles it explicitly — specifically: the `stored` state must be reset when `selectedId` changes, and the "Store locally" button must only appear when `detail.triples` is non-empty AND `detail.has_local_copy === false`.

## What's Solid

- **Upgrade in place is the right call** — one record, origin traceable via `hilo:dataUrl`, no duplicate event IDs
- **`has_local_copy: bool` as an explicit field** rather than deriving from empty triples is clean and avoids ambiguity
- **409 for idempotency** gives the user clear feedback; silent overwrite would hide bugs
- **Re-fetching from local API after import** is the correct confirmation step — it proves the write succeeded, not just that the API call returned 200
- **Reusing `require_jwt`** keeps auth consistent with `GET /events/{id}` — no new auth pattern to introduce
- **Status badge derivation table** is unambiguous and covers all three states cleanly
